# Minimal Academic Website (Hugo)

This repository contains a compact academic personal website scaffold built for Hugo. It follows a simple, text-first style inspired by classic research websites and keeps all audio/video pages under `/lectures/media/` as the single source of truth for podcast and media content.

## What is included

- Top navigation in the requested order: Home, About, Publications, Blog/Notes, Lectures, Podcast, Contact
- Publications index and single pages
- Blog index and single pages
- Tag browsing at `/tags/` and custom tag pages at `/tag/<tag>/`
- Blog archive at `/archive/`
- Lectures index with Series, Standalone, and Media sections
- Lecture series pages at `/lectures/series/<series-slug>/`
- Talk pages at `/lectures/<talk-slug>/`
- Media index at `/lectures/media/`
- Media series pages at `/lectures/media/series/<series-slug>/`
- Episode pages at `/lectures/media/<episode-slug>/`
- Static search page at `/search/` backed by Hugo-generated `index.json`
- RSS outputs for section pages including `/blog/` and `/lectures/media/`
- A Python validation script for required front matter and tag whitelist enforcement
- Starter content: 1 publication, 1 blog post, 1 lecture talk, 1 podcast episode

## Content model

This scaffold uses JSON front matter so the validation script can run without external Python packages.

Required fields:

- `publications`: `title`, `authors`, `year`, `venue`, `type`, `tags`
- `blog posts`: `title`, `date`, `tags`
- `talks`: `title`, `date`, `tags`
- `episodes`: `title`, `date`, `format`, `tags`, `show_notes`

Optional fields are already wired in the templates:

- Publications: `abstract`, `doi`, `url`, `pdf`, `bibtex`
- Blog posts: `summary`
- Talks: `venue`, `abstract`, `materials`, `series`
- Episodes: `embed_url`, `transcript`, `series`

Hugo reserves the top-level `url` key for page routing. In this JSON-front-matter scaffold, store external publication links under `"params": { "url": "https://..." }`; the template still reads that value as the publication URL.

## Local development

1. Install Hugo Extended from the official release page: [https://gohugo.io/installation/](https://gohugo.io/installation/)
2. Validate the content:

```powershell
python .\scripts\validate_content.py
```

3. Run the local development server:

```powershell
hugo server -D
```

4. Open the local site Hugo prints in the terminal, usually `http://localhost:1313/`

## Production build

```powershell
hugo
```

The generated site will be written to `public/`.

## Deploy to GitHub Pages

This repository is configured for deployment with GitHub Pages through GitHub Actions.

1. Create a GitHub repository and push this project to the `main` branch.
2. In GitHub, open `Settings` -> `Pages`.
3. Under `Build and deployment`, set `Source` to `GitHub Actions`.
4. Push any new commit to `main`. The workflow at [`.github/workflows/hugo.yaml`](/C:/Users/fanfa/Documents/codex%20办公/个人网站/.github/workflows/hugo.yaml) will build and publish the site automatically.
5. Open the `Actions` tab in GitHub to watch the first deployment. When it finishes, GitHub will show the live Pages URL.

Notes:

- The workflow uses GitHub's `configure-pages`, `upload-pages-artifact`, and `deploy-pages` actions, which is the current GitHub-recommended deployment flow for Pages.
- The workflow also passes GitHub Pages' computed `baseURL` into Hugo during the build, so the site can work as either a project site or a user site.
- If you later connect a custom domain, update `baseURL` in [config.toml](/C:/Users/fanfa/Documents/codex%20办公/个人网站/config.toml) to that final domain.

Example Git commands after you create the remote repository:

```powershell
git init
git branch -M main
git add .
git commit -m "Initial site"
git remote add origin https://github.com/<your-account>/<your-repo>.git
git push -u origin main
```

## Tag whitelist

Allowed tags are defined in `data/tag-whitelist.json`. Update that file first if you want to introduce a new tag, then add the matching page under `content/tag/` and rerun the validator.

## Notes on search

The `/search/` page uses a static JSON index generated by Hugo. Pagefind was not wired in this scaffold so the site stays dependency-light and can run without a Node install step.
